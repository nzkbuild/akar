# AKAR v0.24.0 — First External Dogfood Trial Report

**Date:** 2026-07-06
**Scope:** use AKAR in advisory mode on one controlled external repo/task and produce an honest dogfood report. No new AKAR features; no AKAR internals refactored. The trial exposes friction and one blocker-class finding (hook templates on external repos) that is recorded here, not fixed in v0.24.
**Method:** built a tiny throwaway Rust fixture outside the AKAR repo, ran the full AKAR advisory loop (doctor → status → preflight --snapshot → request → request --check → governor), read the compiled NEXT_RUN.md, performed the fix manually obeying the NEXT_RUN hard rules, then ran the post-task loop (test → postmortem --diff --baseline → learn --list → governor → doctor). All evidence recorded below verbatim where possible.

---

## 1. Executive verdict

AKAR v0.23.0 **works as an advisory discipline layer on a fresh external repo**, and the Real Doctor correctly diagnoses the environment — but the trial exposed **one genuine blocker and several friction points** that must be addressed before AKAR can be handed to a non-developer user.

The blocker: **`akar hooks --check` and the doctor's hook-template check always FAIL on a fresh external repo**, because the hook templates live in the AKAR source tree (`templates/hooks/pre-tool-call.{sh,ps1}`), not alongside the installed binary. A user who installs `akar.exe` on PATH and runs `akar doctor` in their own repo gets a FAIL and guidance ("Reinstall templates: `akar hooks --install`") that cannot succeed because the install source directory does not exist in their project. This undercuts the doctor's value on the exact use case it was built for.

Everything else worked: the snapshot refused a dirty tree, the baseline was recorded, the compiled NEXT_RUN.md was a high-quality, contract-valid, safety-bounded prompt, the manual fix was a minimal one-line change within budget, and `postmortem --diff --baseline` returned PASS with accurate diff measurement. The advisory loop reduced friction on discipline (budget, baseline, postmortem) and increased friction on setup (bootstrap confusion, missing templates).

**Verdict:** v0.24.0 is a dogfood/audit release. AKAR is closer to "first real try" readiness but **not there yet** — the hook-template discovery blocker must be fixed first (v0.25).

## 5. NEXT_RUN prompt quality

The compiled `.akar/NEXT_RUN.md` was **high quality and contract-valid** (`request --check` PASS on all four checks: sections, minimum content, safety contract, decision consistency).

- **Objective (exact):** "Continue the scoped task without broadening the work."
- **Hard rules:** all 10 present and correct — no git reset/clean/stash/checkout/push, no deleting user files, no modifying Claude config, no retrying blocked commands, no broadening after budget/governor warnings, stop if verification fails.
- **Forbidden commands:** the 10 destructive patterns (git reset/clean/stash/checkout/push, `rm -rf /`, `rm -rf /*`, `sudo rm -rf /`, `del /s /q C:\`, `Remove-Item -Recurse -Force C:\`, broad `git add .`).
- **Stop conditions:** 7, including "Stop if `cargo test` fails."
- **Verification Required:** cargo build --release, cargo test, and the AKAR self-checks.
- **Closing comment:** "AKAR compiled this prompt from local evidence and foundation playbooks. AKAR did NOT run it automatically."

**Quality verdict:** the prompt is a genuinely useful, safety-bounded handoff to Claude. It would let an agent continue correctly without rediscovering basics. This is the strongest part of AKAR's advisory loop. The one gap: the objective is generic ("continue the scoped task") — it does not echo the user's actual task ("fix one small failing test"). The suggested prompt is also generic. This is by design (the governor does not know the user's task text), but a user reading NEXT_RUN cold would not know what task to continue.

## 10. What AKAR helped with

- **Discipline framing:** the baseline → work → postmortem loop made the task's scope explicit and measured. The 3-file/60-LOC budget made "minimal change" a concrete target.
- **Accurate diff measurement:** `postmortem --diff --baseline` correctly reported 1 file / 2 LOC against the budget — exactly the change made, no more.
- **Safety boundaries:** NEXT_RUN.md's hard rules and forbidden-command list would stop a jittery agent from `git reset`ing or broadening the task.
- **Honest diagnostics:** the doctor told the truth about what was missing and what to do. It did not pretend the environment was ready when it wasn't.
- **Contract-valid handoff:** `request --check` PASS gives confidence the compiled prompt is well-formed before handing it to Claude.
- **Governor transitions:** all three decisions (SNAPSHOT_NOW, READY, RUN_POSTMORTEM) matched the real git/baseline state.

## 11. What AKAR made worse

1. **Hook-template blocker (the big one).** `akar hooks --check` and the doctor's hook-template check **always FAIL on a fresh external repo**. The templates live at `templates/hooks/pre-tool-call.{sh,ps1}` relative to the AKAR *source* tree. `hooks::find_hook_templates` searches `<project_root>/templates/hooks/` and `<exe_dir>/templates/hooks/` — neither exists in a user's repo. The doctor's FAIL guidance says "Reinstall templates: `akar hooks --install`", but `--install` copies from the same nonexistent template dir. A fresh-external-repo user hits a FAIL they cannot resolve without the AKAR source tree. This is the single biggest dogfood blocker.
2. **`bootstrap: 0 created, 0 skipped` is confusing.** Bootstrap created the `.akar/` directory (the snapshot then succeeded), but the message reads as "nothing happened." The "templates directory not found" warning explains why 0 templates were copied, but a user reading the top line alone would think bootstrap did nothing.
3. **`Cargo.lock` dirty-tree friction.** `cargo test` generates `Cargo.lock`; `preflight --snapshot` requires a clean tree. A user must commit `Cargo.lock` before snapshotting. AKAR's dirty-tree WARN is correct, but the friction is real on a fresh repo (the user has to decide whether to commit `Cargo.lock`).
4. **Generic NEXT_RUN objective.** The objective ("Continue the scoped task without broadening the work") does not reference the user's actual task. A user reading NEXT_RUN cold does not know what task to continue. The governor does not receive the task text, so this is architectural — but it reduces the prompt's standalone usefulness.
5. **`status` always DEGRADED on fresh external repo.** Because the doctor FAILs on hook templates, `akar status` shows DEGRADED on any repo that hasn't resolved the template issue. Honest, but discouraging for a first-time user.

---

## 12. Confusing or misleading output

- **`bootstrap: 0 created, 0 skipped`** — misleading. Bootstrap created `.akar/` (confirmed: the subsequent snapshot wrote `DIFF_BASELINE.json` into it). The "0 created" counts *template files* copied, not directories created. A user would reasonably conclude bootstrap did nothing.
- **`hooks` help: "AKAR ships hook templates for Claude Code integration."** — true only if the AKAR *source tree* is present. On an installed-binary-only external repo, the templates are not "shipped" in any discoverable location. The help does not say where to find them.
- **`hooks --check` FAIL guidance: "Reinstall templates: `akar hooks --install`."** — the guidance points at a command that cannot succeed on an external repo (the install source dir does not exist). This is the most misleading output in the trial.
- **`doctor` FAIL on hook templates** — technically correct (templates are missing), but it conflates "you haven't installed AKAR's hook templates" with "your environment is broken." For a user who doesn't want the hook, this FAIL is noise.
- **`governor` suggested prompt is generic** ("Continue the scoped task...") — useful as a guardrail, not as a task description. Not misleading, just limited.

## 13. Manual rescue required

One manual rescue was needed: committing `Cargo.lock` to make the working tree clean before `akar preflight --snapshot`. `cargo test` generated `Cargo.lock`; the snapshot requires a clean tree. This is not a bug (the clean-tree requirement is intentional and correct), but it is friction a first-time user would hit. No other rescue was needed — the fix itself was trivial and within budget.

No hook safety evidence was collected: the PreToolUse hook was not active in the fixture (templates missing), so `HOOK_EVENTS.jsonl` stayed empty and no BLOCK/ALLOW evidence was generated. The hook-blocking behavior was therefore not exercised in this trial. This is a direct consequence of the hook-template blocker (§11.1) — the one piece of real runtime enforcement AKAR provides could not be tested on the external repo.

## 14. Dogfood readiness verdict

**Not yet ready for a non-developer "first real try."** AKAR v0.23.0 is honest, safe, and the advisory loop (baseline → budget → postmortem) works correctly on an external repo. But three things block hand-off to a non-developer user:

1. **Hook templates are not discoverable from an installed binary.** A fresh-external-repo user gets a FAIL they cannot resolve without the AKAR source. This must be fixed (embed templates in the binary, or ship them alongside it, or have `hooks --install` fetch from a known location) before v1.0.
2. **`bootstrap` output is confusing.** "0 created" when it created `.akar/` misleads.
3. **`status` is DEGRADED on every fresh external repo** until the hook-template issue is resolved.

For a developer who already understands AKAR, the advisory loop is genuinely useful and the postmortem measurement is accurate. For a non-developer, the setup friction and the unresolvable FAIL would stop them.

**Partial readiness:** the discipline loop (snapshot → work → postmortem) is ready. The setup path (bootstrap → hooks) is not.

## 15. Required fixes before next dogfood

Ordered by impact:

1. **(Blocker) Make hook templates discoverable from the installed binary.** Either embed `pre-tool-call.{sh,ps1}` in the binary via `include_str!` and have `hooks --install` write them out, or ship them in a known location next to the binary. `hooks --check` and the doctor should then PASS on a fresh external repo after `hooks --install`. This is the v0.25 priority.
2. **Fix `bootstrap` output.** Distinguish "created `.akar/` directory" from "copied N template files." A user must be able to tell bootstrap did something. Low effort, high clarity.
3. **Soften the doctor's hook-template check.** Consider making missing hook templates a WARN (advisory — the hook is optional) rather than a FAIL (dogfood-stop), **or** keep it FAIL but only after templates are genuinely installable from the binary. Today it FAILs unconditionally on external repos, which is the worst combination.
4. **Echo the user's task in NEXT_RUN.** `request` could accept and thread the task prompt into the compiled NEXT_RUN objective/suggested-prompt, so a cold reader knows what to continue. Architectural, but high value.
5. **Document the `Cargo.lock` friction.** Either `preflight --snapshot` should tolerate an untracked `Cargo.lock` (generated artifact), or `init`/`bootstrap` should pre-commit it / add it to `.gitignore` guidance. Minor.

None of these are addressed in v0.24 (dogfood/audit release only, no new features). They are the agenda for v0.25.

## 16. Honest conclusion

AKAR v0.24.0 is a **dogfood/audit release** that proves the advisory loop works on a real external repo and exposes exactly where it breaks. The good news is substantial: the Real Doctor diagnoses the environment correctly, the baseline snapshot refuses a dirty tree, the compiled NEXT_RUN.md is a contract-valid, safety-bounded handoff, the manual fix stayed within budget, and `postmortem --diff --baseline` measured the change accurately and returned PASS. The advisory-only posture held throughout — AKAR never executed, never mutated git, never modified Claude settings, never installed hooks, never auto-applied anything.

The bad news is concrete and now on the record: **the hook templates are not discoverable from an installed binary**, so every fresh external repo gets an unresolvable doctor FAIL and `status: DEGRADED`. This is the one thing standing between AKAR and a non-developer "first real try." It is a setup-path defect, not a discipline-path defect — the discipline path is ready.

This trial also confirmed the v0.21 audit's #1 dogfood risk (the `run`/`mission` expectation gap) is mitigated by the v0.22 advisory banners: a user reading the banner cannot reasonably expect AKAR to execute. No new expectation gap appeared.

v0.24.0 changes no AKAR runtime behavior. It records the trial honestly so v0.25 can fix the blocker.

## 17. Next recommended release

**v0.25.0 — Hook Template Discovery.** Make the PreToolUse hook templates installable from the installed binary (embed via `include_str!`, write out on `hooks --install`), so `hooks --check` and the doctor PASS on a fresh external repo after install. Fix the `bootstrap` "0 created" confusion. Reconsider whether missing hook templates should be WARN (advisory) or FAIL. Re-run this dogfood trial on a fresh external repo and confirm the doctor reaches OK/WARN without the source tree.

**v0.26.0 — NEXT_RUN task threading.** Accept the user's task prompt in `request` and thread it into the compiled objective/suggested-prompt so NEXT_RUN is self-describing.

**v1.0 design review** — after v0.25 + v0.26 + a second external-repo dogfood trial (ideally non-Rust) confirms the setup path is friction-free. The review decides whether v1.0 stays advisory-only (recommended) or authorizes a bounded execution path.
