# AKAR v0.48.0 — AI-Facing Delivery Design

## 1. Executive Verdict

**CLAUDE.md managed snippet is the correct primary delivery mechanism. Manual fallback
is the correct backup.** After deep-evaluating 7 mechanisms against 16 requirements
across 4 categories (functional, safety, trust, portability), the CLAUDE.md snippet
is the only mechanism that satisfies all requirements without crossing AKAR's
advisory boundary. It delivers AKAR context into the AI session at session-start
(zero user action after setup), preserves AKAR's "never executes project code"
guarantee, is opt-in with explicit user confirmation, is clearly delimited and
removable, and works within documented Claude Code behavior.

The manual fallback (AI reads `.akar/NEXT_RUN.md` from disk) remains available for
all other scenarios — projects without CLAUDE.md, users who decline the snippet,
and non-Claude-Code AI tools.

This report does not implement anything. It provides the evidence-based design that
a future implementation release will follow. No src/ changes, no CLI behavior changes,
no config.toml, no enable/disable commands, no auto-run. Design only.

### 4.3 Key Observation

NEXT_RUN.md is the only artifact the AI **must** read to receive AKAR's
discipline-tuned guidance. The memory files (PROJECT_DNA, STATE, VERIFY_RECIPE,
etc.) are supplementary — nice to have but not essential to the per-task
discipline loop. The delivery design should focus on NEXT_RUN.md as the

Most memory files in this repo contain template/placeholder content — they exist
but aren't actively maintained. This is honest: the project's own dogfood shows
that memory files require discipline to keep current, and the delivery mechanism
should not depend on them being populated.

## 5. The Delivery Problem Defined

### 5.1 Current State

After `akar prepare "<task>"`, the user has a fully compiled NEXT_RUN.md at
`.akar/NEXT_RUN.md`. This file contains everything the AI needs: task objective,
budget limits, allowed/forbidden commands, stop conditions, verification steps,
governor decision, and hard rules. It is the best discipline artifact AKAR
produces.

But it sits on disk. The AI does not automatically know it exists. The user must
either:
1. Copy/paste NEXT_RUN.md content into the Claude Code prompt
2. Tell Claude "read .akar/NEXT_RUN.md"
3. Hope Claude discovers it during filesystem exploration

In v0.45.0 dogfood, Claude successfully read NEXT_RUN.md when told to — proving
the file-on-disk approach works. But the user must remember to tell Claude, and
that "must remember" step is exactly the manual-relay bottleneck.

### 5.2 The Gap

```
User runs: akar prepare "fix the login button"
           ↓
  NEXT_RUN.md written to .akar/  ← AKAR's job is done
           ↓
  [GAP: how does this reach the AI?]  ← v0.48.0 design problem
           ↓
  Claude Code session starts
           ↓
  Claude reads NEXT_RUN.md  ← desired outcome
```

The gap is approximately 0 commands when it works (Claude discovers the file)
and 1 user action when it doesn't (user says "read .akar/NEXT_RUN.md"). The
v0.45.0 evidence shows it works when the user remembers. The v0.44.0 design
identified CLAUDE.md snippet as the mechanism to close this gap.

### 5.3 Why This Matters

Without automatic delivery, AKAR is an enhancement layer that requires manual
relay to enhance anything. The user is the bridge between the discipline layer
and the worker. Close this gap, and AKAR transitions from "manual checklist" to
"enhancement layer" — the single most important architectural transition in
AKAR's path to the North Star.

### 5.4 What Success Looks Like

After `akar prepare`, the user starts Claude Code. Claude reads NEXT_RUN.md
without being told. The task proceeds with AKAR's discipline guidance present
in the AI's context from the start. Zero user relay actions.

## 6. Requirements: Functional

| # | Requirement | Rationale |
|---|---|---|
| F1 | AI receives AKAR context without user copy/paste or verbal relay | The manual-relay step is the bottleneck being eliminated |
| F2 | Context is available at or near session start | Late delivery means the AI works without guidance for initial messages |
| F3 | Context includes at minimum: task, budget, verification command, stop conditions, governor decision | These are the minimum guardrails from NEXT_RUN.md |
| F4 | Full NEXT_RUN.md remains readable on disk | The AI can access complete detail when needed |

| F6 | User action to enable delivery is a one-time setup per project | Not a per-session or per-task action |
| F7 | Delivery does not require the user to modify AI tool configuration manually | AKAR should handle the mechanics, user only confirms intent |

| # | Requirement | Rationale |
|---|---|---|

| S2 | AKAR never modifies Claude Code settings files | `~/.claude/settings.json` is user-owned territory |
| S3 | AKAR never calls model APIs | AKAR is not an AI agent |
| S4 | AKAR never commits, pushes, resets, or mutates git state | Core safety guarantee — unchanged |
| S5 | Delivery mechanism does not auto-execute anything | The AI reads context; context does not run commands |
| S6 | Delivery does not add hidden or obscured instructions | User can see and audit exactly what AKAR added |

| # | Requirement | Rationale |
|---|---|---|

| # | Requirement | Rationale |
|---|---|---|
| P1 | Manual fallback works on any AI tool with filesystem access | Not all tools have CLAUDE.md |

## 10. Mechanism A: Managed CLAUDE.md Snippet (Primary Candidate)

### How It Works

During `akar init` (future enhancement, not implemented here), AKAR offers to
append a clearly delimited section to the project's CLAUDE.md. The section
instructs Claude to read `.akar/NEXT_RUN.md` before starting coding tasks.
CLAUDE.md is loaded into Claude's system prompt at session start, so the
instruction is always present.

### Functional Evaluation

| Req | Status | Evidence |
|---|---|---|
| F1 | **Satisfied** | Claude reads CLAUDE.md at session start automatically; the instruction "read .akar/NEXT_RUN.md" is in the system prompt; Claude reads the file without user relay |
| F2 | **Satisfied** | CLAUDE.md is loaded at session start — context instruction is present from message 1 |
| F3 | **Satisfied** | Snippet points to full NEXT_RUN.md which contains all required fields |
| F4 | **Satisfied** | Snippet is a pointer, not a replacement; NEXT_RUN.md remains on disk |
| F5 | **Satisfied** | CLAUDE.md auto-loading is documented Claude Code behavior; no speculation needed |
| F6 | **Satisfied** | One-time `akar init` (or `akar init --claude`); never per-session |
| F7 | **Satisfied** | AKAR appends the snippet; user only confirms |

### Safety Evaluation

| Req | Status | Evidence |
|---|---|---|
| S1 | **Satisfied** | Writing markdown to a project file is not executing project code |
| S2 | **Satisfied** | CLAUDE.md is a project file, not a Claude Code settings file |
| S3 | **Satisfied** | No API calls involved |
| S4 | **Satisfied** | AKAR writes to CLAUDE.md; git commit is the user's decision |
| S5 | **Satisfied** | The snippet says "read a file" — it is an instruction to the AI, not an execution command |
| S6 | **Satisfied** | The snippet is visible in CLAUDE.md, version-controlled, auditable |

### Trust Evaluation

| Req | Status | Evidence |
|---|---|---|
| T1 | **Satisfied** | AKAR asks "Add AKAR session guidance to CLAUDE.md? [y/N]" during init; snippet is never added without confirmation |
| T2 | **Satisfied** | Delimited with `## AKAR Session Guidance (managed by akar init)` header and `<!-- AKAR section ends -->` footer |
| T3 | **Satisfied** | User opens CLAUDE.md, deletes the AKAR section, saves — delivery is gone |
| T4 | **Satisfied** | AKAR uses delimiter markers to identify its section; replaces only its own block on update; never touches other content |

### Portability Evaluation

| Req | Status | Evidence |
|---|---|---|
| P1 | **Satisfied** (via fallback) | Non-Claude-Code tools fall back to manual file read; the file is still on disk |
| P2 | **Satisfied** | Writing a text file is platform-independent |
| P3 | **Satisfied** | CLAUDE.md is a stable convention; even if the loading mechanism changes, the file is still there for the AI to read manually |

### Unique Advantages

- **Zero per-session user action.** Setup once, works forever.
- **Zero token waste from repetition.** CLAUDE.md is in the system prompt, not repeated per message.
- **Works within documented behavior.** No speculation about hook visibility or slash command auto-fire.
- **User has full visibility.** The snippet is in a project file they already manage.
- **Removable in seconds.** Delete the section, delivery stops.
- **Version-controlled.** The snippet travels with the project; teammates who clone get it.

### Risks and Mitigations

| Risk | Severity | Mitigation |
|---|---|---|
| User doesn't run `akar init` (existing projects) | Medium | `akar prepare` output reminds user: "context: CLAUDE.md not configured. Run `akar init --claude` to enable automatic context delivery." |
| CLAUDE.md already has AKAR content from manual editing | Low | Delimiter-based detection; AKAR only touches content between its markers |
| User edits CLAUDE.md and accidentally breaks the snippet | Low | Snippet is simple markdown; hard to break; `akar init --claude` can repair |
| Claude ignores the instruction (model non-compliance) | Low | Claude reads CLAUDE.md instructions reliably; this is the entire premise of CLAUDE.md |
| Snippet grows stale (points to wrong AKAR version) | Low | Snippet is version-agnostic ("read .akar/NEXT_RUN.md"); doesn't embed AKAR version |

## 11. Mechanism B: Project-Local Instruction File

### How It Works

Instead of modifying CLAUDE.md, AKAR writes a separate instruction file (e.g.,
`.akar/CLAUDE.md` or `AKAR.md` at project root) that contains the same "read
NEXT_RUN.md" instruction. Claude would need to discover and read this file.

### Evaluation

This is mechanism A without the auto-loading advantage. CLAUDE.md is special
because Claude Code loads it into the system prompt automatically. Any other
filename requires the AI to discover it via filesystem exploration or user
prompting — which is exactly the manual-relay problem we're trying to solve.

**Verdict: Rejected.** Adds a file without adding auto-delivery. Mechanism A
achieves the same goal with the critical advantage of auto-loading.

## 12. Mechanism C: Claude Slash Command

### How It Works

AKAR installs a project-level slash command (e.g., `/akar:context`) that reads
and summarizes NEXT_RUN.md. User types `/akar:context` at session start.

### Evaluation

| Requirement | Status |
|---|---|
| F1 (no manual relay) | **Failed** — user must type the slash command |
| F6 (one-time setup) | **Satisfied** — command is installed once |

**Verdict: Accepted as optional supplement, rejected as primary.** A slash command
is better than copy/paste but still requires per-session user action. It fails the
core functional requirement F1. It could serve as an alternative for users who
decline the CLAUDE.md snippet but want something better than manual relay.

## 13. Mechanism D: Hook-Mediated Reminder

### How It Works

The existing PreToolUse hook detects the first tool call in a session and writes
a reminder or attempts to surface NEXT_RUN.md content through hook output.

### Evaluation

| Requirement | Status |
|---|---|
| F5 (documented behavior) | **Failed** — hook output visibility to the model is not documented |

The v0.44.0 report flagged this: "Hook stdout/stderr is not guaranteed to be
injected into the model's context by Claude Code." This remains true. Without
proof that hook output reaches the AI, this mechanism is built on a shaky
foundation.

**Verdict: Postponed pending hook visibility testing.** If future investigation
proves hook output is model-visible, this becomes a powerful mechanism for
mid-session context injection. Until then, it cannot be the primary mechanism.

## 14. Mechanism E: Shell Wrapper / Launcher

### How It Works

User runs `akar-claude "<task>"` instead of `claude`. The wrapper runs prepare,
then launches Claude Code.

### Evaluation

| Requirement | Status |
|---|---|
| P2 (cross-platform) | **Failed** — wrapper is shell-specific (bash vs PowerShell) |
| P3 (resilient to Claude changes) | **Failed** — fragile dependency on launch mechanism |
| Trust (transparency) | **Failed** — hides AKAR behavior behind opaque launcher |

The v0.44.0 report rejected this: "Too fragile, too platform-specific, too much
magic." Nothing has changed.

**Verdict: Rejected.** The wrapper approach contradicts AKAR's transparency
principle and adds fragility without benefit over the CLAUDE.md snippet.

## 15. Mechanism F: File Watcher / Daemon

### How It Works

AKAR runs as a background process watching for repo changes, prompting the user
or writing context files when it detects session-like activity.

### Evaluation

| Requirement | Status |
|---|---|
| S1 (no project execution) | **Satisfied** — daemon only watches files |
| P2 (cross-platform) | **Failed** — daemon lifecycle differs significantly across platforms |
| Complexity | **Excessive** — persistent process for a problem solved by a text file |

The v0.44.0 report rejected this: "Too complex, too risky, too platform-specific."

**Verdict: Rejected.** A persistent daemon to solve a problem that a 10-line
markdown snippet solves is unjustifiable complexity. AKAR is a fire-and-exit CLI,
not a background service.

## 16. Mechanism G: Manual Fallback

### How It Works

No delivery mechanism. NEXT_RUN.md is on disk. User tells Claude to read it, or
Claude discovers it. This is the current state and the v0.44.0 mechanism A.

### Evaluation

| Requirement | Status |
|---|---|
| F1 (no manual relay) | **Failed** — user must relay or AI must discover |
| All safety requirements | **Satisfied** — no mechanism means no safety risk |
| All trust requirements | **Satisfied** — no mechanism means no trust concern |
| All portability requirements | **Satisfied** — manual always works |

**Verdict: Keep as universal fallback.** Manual mode is the safest, most portable
option. It works on every AI tool, every platform, every scenario. It fails the
core functional requirement (eliminating manual relay), which is why it is the
fallback, not the primary.

## 18. Recommendation

### Primary: Managed CLAUDE.md Snippet (Mechanism A)

Add a clearly delimited section to the project's CLAUDE.md during `akar init`
(with explicit user confirmation). The section instructs Claude to read
`.akar/NEXT_RUN.md` before starting coding tasks.

### Fallback: Manual File Read (Mechanism G)

For projects without CLAUDE.md, users who decline the snippet, and non-Claude-Code
AI tools, NEXT_RUN.md remains on disk. The AI reads it when the user asks or when
it discovers it. `akar prepare` output includes a reminder when CLAUDE.md is not
configured.

### Supplementary: Slash Command (Mechanism C)

Optional. For users who want a middle ground — better than copy/paste, less
integrated than CLAUDE.md snippet. A `/akar:context` slash command that reads and
summarizes NEXT_RUN.md. Not required for the primary flow.

### Postponed: Hook-Mediated Reminder (Mechanism D)

Revisit after testing whether Claude Code surfaces hook output to the model. If
yes, hook-mediated injection becomes a powerful complement for mid-session context
updates.

### Why Not Two Primaries

Dual primary mechanisms introduce ambiguity about which one is active, increase
the surface area for bugs, and complicate the user's mental model. One primary
(CLAUDE.md snippet) with one fallback (manual) is the simplest correct design.

## 19. CLAUDE.md Snippet Text Design

### Exact Snippet Text

```markdown
## AKAR Session Guidance (managed by `akar init`)

Before starting any coding task, read `.akar/NEXT_RUN.md`. It contains:
- The current task objective and scope
- Budget limits (files and lines of code)
- Allowed and forbidden commands
- Required verification steps
- Stop conditions
- The loop governor's decision

After completing work, verify you stayed within the budget and followed the
stop conditions. The user will run `akar finish` to measure the diff.

<!-- AKAR section ends -->
```

### Design Rationale

**Header:** `## AKAR Session Guidance (managed by \`akar init\`)` — clearly
identifies the section as AKAR-managed, not user-written. The "managed by"
notation tells the user that AKAR may update this section (during future
`akar init` runs) and that manual edits may be overwritten.

**Instruction:** "Before starting any coding task, read `.akar/NEXT_RUN.md`" —
direct, unambiguous. Does not say "consider reading" or "you may want to read."
It is an instruction to read the file, not to execute anything.

**Bullet list of contents:** Tells Claude what it will find in NEXT_RUN.md. This
serves two purposes: (1) it sets expectations so Claude knows why it's reading
the file, and (2) if for some reason Claude cannot read the file (permission error,
missing file), it still knows the categories of guidance it should seek.

**Post-work instruction:** "verify you stayed within the budget" — closes the
loop. Claude should self-check against the budget before the user runs `akar finish`.

**Footer:** `<!-- AKAR section ends -->` — HTML comment delimiter invisible in
rendered markdown but machine-parseable. AKAR uses this to find its section for
idempotent updates (replace everything between the header and footer).

### What the Snippet Does NOT Include

- **Does not inline task details.** Task details go in NEXT_RUN.md, which changes
  per task. The snippet is stable project configuration.
- **Does not include AKAR version number.** The snippet is version-agnostic. It
  points to `.akar/NEXT_RUN.md` regardless of AKAR version.
- **Does not include budget numbers or verification commands.** Those are
  per-task and belong in NEXT_RUN.md, not the stable snippet.
- **Does not reference specific AI models or capabilities.** The snippet works
  for any Claude model.
- **Does not include "always" or "never" absolutes beyond the core instruction.**
  The instruction is strong ("read the file") but bounded (one specific file).

### Idempotent Update Strategy

When `akar init` (or `akar init --claude`) runs on a project that already has
the AKAR snippet:

1. Find the header `## AKAR Session Guidance (managed by \`akar init\`)` in CLAUDE.md
2. Find the footer `<!-- AKAR section ends -->`
3. Replace everything between them (inclusive) with the current snippet
4. If header not found: append snippet at end of CLAUDE.md
5. If CLAUDE.md doesn't exist: create it with the snippet

This ensures AKAR's snippet stays current without duplicating or leaving orphaned
sections. User content outside the delimiters is never touched.

## 20. Context Size Tier Design

### Tier Definitions

| Tier | Size | Content | Delivery | Use Case |
|---|---|---|---|---|
| **Tiny** | 7 lines | Task, project kind, budget, verification command, governor decision, stop rule | Inlined in CLAUDE.md snippet (always in system prompt) | Every session — zero-cost guardrails |
| **Normal** | ~30 lines | Tiny + allowed commands, forbidden commands, stop conditions, verification hints | Not in CLAUDE.md; AI reads from NEXT_RUN.md on disk | Complex tasks needing full constraint set |
| **Full** | ~110 lines | All 11 sections of NEXT_RUN.md | On disk only | Detailed reference when needed |

### Tiny Context Design (7 lines, inlined in CLAUDE.md snippet)

The v0.44.0 report proposed a tiny context tier for inline inclusion in the
CLAUDE.md snippet. After re-evaluation, **the tiny context should NOT be inlined
in the CLAUDE.md snippet.** Rationale:

1. **Tiny context is per-task.** It changes every time the user runs
   `akar prepare`. CLAUDE.md is project-stable configuration, not per-task state.
2. **Updating CLAUDE.md per task is invasive.** It means AKAR writes to CLAUDE.md
   on every `akar prepare`, which crosses the line from "init-time setup" to
   "runtime modification of project files."
3. **CLAUDE.md is version-controlled.** Per-task updates would create a dirty
   git tree on every prepare — exactly the friction point we're trying to eliminate.
4. **The snippet instruction already tells Claude to read NEXT_RUN.md.** Adding
   the tiny context inline is redundant — Claude will read the file and get all
   the detail.

**Revised design:** The CLAUDE.md snippet contains only the stable instruction
(10 lines). It tells Claude to read NEXT_RUN.md. Claude reads NEXT_RUN.md, which
contains all tiers of context (tiny, normal, and full are all sections of the
same file). No per-task CLAUDE.md modification needed.

### When Claude Reads NEXT_RUN.md

The flow:
1. Session starts → CLAUDE.md is loaded into system prompt
2. System prompt contains: "Before starting any coding task, read `.akar/NEXT_RUN.md`"
3. Claude uses the Read tool to read NEXT_RUN.md
4. Claude now has all context tiers (tiny through full) from the single file read
5. Claude proceeds with the task, guided by budget, stop conditions, verification

**Token cost:** One file read per session. NEXT_RUN.md is ~110 lines (~2-3K tokens).
This is negligible compared to the token waste of the user pasting context into
every prompt or the AI working without guidance and making budget-exceeding changes.

### Why the v0.44.0 Tiny-Inline Design Changed

The v0.44.0 design proposed inlining a 7-line tiny context in the CLAUDE.md
snippet itself. That design assumed the tiny context would be static enough to
warrant CLAUDE.md inclusion. The per-task nature of the context (different task,
budget, verification per prepare) makes inlining impractical without per-task
CLAUDE.md writes — which violates trust requirement T4 (AKAR only modifies its
section during init, not during runtime).

This v0.48.0 design simplifies: the snippet is purely a pointer. It never changes
after init. All task context stays in NEXT_RUN.md where it belongs.

## 21. Toggle Mechanism Design

### Design Constraint

This report does not implement any toggle mechanism. No `akar enable`, no
`akar disable`, no config.toml. The toggle design here is for a future
implementation release.

### Recommended Approach: Presence of Snippet = Enabled

The simplest toggle: if the AKAR section exists in CLAUDE.md, delivery is enabled.
If it doesn't, delivery is disabled.

**Enable:** Run `akar init --claude` → snippet is added → delivery enabled.
**Disable:** Delete the AKAR section from CLAUDE.md → delivery disabled.
**Status check:** `akar doctor` and `akar status` detect whether the snippet is
present in CLAUDE.md and report it.

### Why Not config.toml

The v0.44.0 report proposed a `.akar/config.toml` with `[auto]` settings and
`akar enable`/`akar disable` commands. This is unnecessarily complex for a
single boolean (is the snippet there or not). Adding a config file format, a
parser, and two new CLI commands to manage one setting is over-engineering.

The snippet's presence in CLAUDE.md IS the configuration. No separate config
file needed. No commands needed beyond `akar init --claude` (which already
exists in the design). The user can see the state by looking at CLAUDE.md or
running `akar doctor`/`akar status`.

### Future-Proofing

If future AKAR versions add more auto-behaviors (auto-snapshot on prepare, etc.),
a config file may become warranted. But for v0.48.0's scope — exactly one delivery
mechanism with exactly one boolean state — the snippet's presence is sufficient.

### Visibility

`akar doctor` and `akar status` should report whether AI-facing delivery is active.
Proposed output:

```
akar status
  AKAR v0.48.0 | project: akar | Rust
  state: HEALTHY, READY
  delivery: active (CLAUDE.md configured)  ← new line
```

```
akar status (without snippet)
  AKAR v0.48.0 | project: akar | Rust
  state: HEALTHY, READY
  delivery: inactive (CLAUDE.md not configured — run 'akar init --claude')
```

## 22. Failure Modes

### What Can Go Wrong and What Happens

| Failure | Impact | Detection | Recovery |
|---|---|---|---|
| CLAUDE.md doesn't exist | `akar init --claude` creates it with the snippet | Detected during init | Create file with snippet |
| CLAUDE.md exists but AKAR section was manually deleted | Delivery stops silently | `akar doctor` reports delivery inactive | Run `akar init --claude` to re-add |
| User edits AKAR section, breaks formatting | Claude may misinterpret the instruction | Hard to detect automatically | Run `akar init --claude` to restore canonical text |
| NEXT_RUN.md doesn't exist when Claude tries to read it | Claude reports file not found; continues without AKAR guidance | Claude surfaces the error | User runs `akar prepare "<task>"` |
| NEXT_RUN.md exists but is stale (from previous task) | Claude reads old task context; works on wrong task | User should notice task mismatch | Run `akar prepare "<new task>"` before starting |
| Claude ignores the CLAUDE.md instruction | No AKAR context; task proceeds unguided | User observes Claude didn't mention AKAR | Tell Claude "read .akar/NEXT_RUN.md" (manual fallback) |
| CLAUDE.md is in .gitignore or not tracked | Snippet works locally but doesn't travel with repo | `akar doctor` may warn if CLAUDE.md is gitignored | User decides: commit CLAUDE.md or accept local-only delivery |
| Multiple projects share a CLAUDE.md (monorepo) | Snippet points to wrong `.akar/` | Confusion if multiple .akar/ directories exist | Each sub-project needs its own CLAUDE.md or the snippet must use relative paths correctly |
| AKAR snippet header is translated/localized | AKAR can't find its section for idempotent update | `akar init --claude` appends duplicate section | Use stable English header; never localize the delimiter |

### Silent Failure Mitigation

The worst failure mode is silent: the snippet is present but Claude doesn't read
NEXT_RUN.md, and the user doesn't notice. Mitigations:

1. **`akar status` reports delivery state** so the user can verify before starting.
2. **`akar prepare` output includes a delivery status line** — "context: CLAUDE.md
   configured — Claude will read .akar/NEXT_RUN.md automatically" or "context:
   CLAUDE.md not configured — tell Claude to read .akar/NEXT_RUN.md".
3. **NEXT_RUN.md includes the task description** — if Claude reads it, the task
   context will be visible in Claude's response.

### The User Is the Ultimate Safety Net

AKAR's delivery is advisory. If the snippet fails, if Claude ignores it, if
NEXT_RUN.md is missing — the user can always say "read .akar/NEXT_RUN.md." The
manual fallback is always available. This is by design: AKAR enhances, it doesn't
control.

## 23. Implementation Scope (For Future Release)

This section defines what a future implementation release would build. It is not
a commitment — it is a scope boundary to prevent implementation drift.

### What Would Be Built

1. **`akar init --claude` flag** — when passed, unconditionally adds/updates the
   AKAR snippet in CLAUDE.md (with idempotent replace logic)
2. **`akar init` interactive prompt** — when CLAUDE.md exists, offers to add the
   snippet: "Add AKAR session guidance to CLAUDE.md? [y/N]"
3. **Snippet management functions** — `find_akar_section()`, `write_akar_snippet()`,
   `remove_akar_section()` in a new `claude_md.rs` module
4. **`akar doctor` delivery check** — new check: "CLAUDE.md delivery: active/inactive"
5. **`akar status` delivery line** — new line in status output showing delivery state
6. **`akar prepare` delivery reminder** — when snippet is absent, output includes
   a reminder that manual relay is needed
7. **Tests** — for snippet insert, idempotent replace, section detection, removal,
   CLAUDE.md-not-found, empty CLAUDE.md, snippet already present

### What Would NOT Be Built

- No `akar enable` / `akar disable` commands (presence of snippet is the toggle)
- No `.akar/config.toml` (not needed for a single boolean)
- No auto-run, auto-invocation, auto-execution (Layer 3+ concern)
- No per-task CLAUDE.md modification (snippet is stable; task context is in NEXT_RUN.md)
- No hook-mediated injection (postponed pending hook visibility testing)
- No slash command installation (supplementary, not in v0.48 scope)
- No CLAUDE.md creation without user confirmation (opt-in always)

### Hard Safety Boundaries (Unchanged)

- AKAR never executes project code
- AKAR never modifies Claude Code settings files
- AKAR never calls model APIs
- AKAR never commits, pushes, resets, or mutates git state
- AKAR never modifies CLAUDE.md content outside its delimited section
- AKAR never adds the snippet without explicit user confirmation

## 24. Rejected Approaches

| Approach | Why Rejected |
|---|---|
| Inlining tiny context in CLAUDE.md (v0.44.0 proposal) | Per-task context changes on every prepare; CLAUDE.md is project-stable config; per-task writes to CLAUDE.md create dirty-tree friction |
| `.akar/config.toml` for snippet toggle | Unnecessary complexity for a single boolean; snippet presence in CLAUDE.md IS the configuration |
| `akar enable` / `akar disable` commands | Additional CLI surface for something `akar init --claude` and manual deletion already handle |
| Writing CLAUDE.md without confirmation | Violates trust requirement T1 (opt-in); AKAR never modifies project files without asking |
| Auto-detecting session start and injecting context | Requires daemon or hook mechanism; both rejected for complexity; CLAUDE.md snippet is simpler |
| Multiple simultaneous delivery mechanisms | Increases surface area, ambiguous state ("which mechanism is active?"), harder to debug |

## 25. Comparison to v0.44.0 Design

This v0.48.0 design refines the v0.44.0 design with evidence gathered in v0.45-0.47:

| Aspect | v0.44.0 Design | v0.48.0 Design | Why Changed |
|---|---|---|---|
| Tiny context location | Inlined in CLAUDE.md snippet (7 lines) | Not inlined; stays in NEXT_RUN.md | Per-task context changes; CLAUDE.md is stable; per-task writes cause dirty-tree friction |
| Toggle mechanism | `.akar/config.toml` + `akar enable`/`akar disable` | Snippet presence in CLAUDE.md | One boolean doesn't need a config file, parser, and two CLI commands |
| Snippet update timing | During `akar init` only | Same | Confirmed correct — no per-task CLAUDE.md writes |
| Primary mechanism | CLAUDE.md snippet | Same | Confirmed — no other mechanism passes all requirements |
| Fallback | Manual (project file on disk) | Same | Confirmed — manual always works |
| Slash command | Intermediate Layer 1 solution | Supplementary, not required | Prepare/finish (v0.46) already solved the Layer 1 problem; slash command is now optional |
| Hook-mediated | Postpone — test first | Same | Hook output visibility still unverified |
| Shell wrapper | Rejected | Same | Still too fragile |
| File watcher | Rejected | Same | Still too complex |

The v0.44.0 design was 90% correct. The two refinements (no inline tiny context,
no config.toml toggle) simplify the design without losing capability.

## 26. Honest Conclusion

AKAR v0.47.0 proved that prepare/finish reduces manual commands from 8 to 2
across all five project lanes. The remaining bottleneck is AI-facing delivery:
NEXT_RUN.md sits on disk and the user must manually bridge it into the AI session.

This v0.48.0 design defines the bridge. A 10-line managed snippet in CLAUDE.md
that says "read `.akar/NEXT_RUN.md` before starting" is the smallest mechanism
with the highest leverage. It satisfies all 16 requirements across functional,
safety, trust, and portability categories. It preserves AKAR's advisory boundary
completely. It is opt-in, auditable, removable, and version-controlled. It works
within documented Claude Code behavior — no speculation, no undocumented APIs.

The design intentionally rejects complexity: no config file for a single boolean,
no enable/disable commands for something `akar init --claude` already handles,
no inline tiny context that would require per-task CLAUDE.md writes. The snippet
is a stable pointer; all task context stays in NEXT_RUN.md.

What this design does NOT do: implement anything. No src/ changes. No CLI
behavior changes. No CLAUDE.md modification. This is a design report. The
implementation belongs in a future release after this design is reviewed and
accepted.

The path from here:
- v0.48.0 (this report): design the delivery mechanism
- Future v0.49.0: implement CLAUDE.md snippet management in `akar init`
- Future v0.50.0: dogfood the full AI-facing delivery loop with measurements

If this design is accepted, AKAR will have a complete, designed path from manual
CLI toolkit (Layer 0) through friction-reduced advisory (Layer 1, done) through
AI-facing delivery (Layer 2, designed here). The North Star's "enhancement layer"
is not built yet — but the blueprint is complete.
