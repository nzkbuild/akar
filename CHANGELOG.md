## v0.24.0 — 2026-07-06
First External Dogfood Trial. Dogfood/audit release only — no new AKAR features, no internals refactored, no runtime behavior changed. Used AKAR v0.23.0 in advisory mode on a tiny throwaway Rust fixture (`../akar-dogfood-fixture`, outside the AKAR repo) with one buggy `add()` function and one failing test. Ran the full advisory loop: `akar doctor` → `status` → `bootstrap` → `preflight --snapshot` → `request` → `request --check` → `governor --json --no-exit-code`; read the compiled `.akar/NEXT_RUN.md`; performed the fix manually obeying the NEXT_RUN hard rules (minimal one-line change `a * b` → `a + b`, within the 3-file/60-LOC Bugfix budget); then `cargo test` (PASS) → `postmortem --diff --baseline` (PASS, 1 file / 2 LOC) → `learn --list` (0 patches) → `governor` (RUN_POSTMORTEM) → `doctor`. Added `docs/audits/AKAR_V0_24_FIRST_EXTERNAL_DOGFOOD_REPORT.md` (17 sections). Trial verdict: the advisory discipline loop (baseline → budget → postmortem) works correctly on an external repo and the Real Doctor diagnoses the environment honestly, BUT one blocker was exposed — `akar hooks --check` and the doctor's hook-template check always FAIL on a fresh external repo because the hook templates live in the AKAR source tree (`templates/hooks/pre-tool-call.{sh,ps1}`), not alongside the installed binary, and `hooks --install`'s FAIL guidance points at a command that cannot succeed without the source tree. Friction findings: `bootstrap: 0 created, 0 skipped` is misleading (it created `.akar/` but the message counts template files); `preflight --snapshot` requires a clean tree so `cargo test`-generated `Cargo.lock` had to be committed first; the NEXT_RUN objective is generic and does not echo the user's actual task; `akar status` shows DEGRADED on every fresh external repo due to the hook-template FAIL. No hook safety evidence was collected (the hook could not be activated in the fixture — templates missing). Required fixes recorded for v0.25: (1) make hook templates discoverable from the installed binary (embed via `include_str!`, write on `hooks --install`); (2) fix `bootstrap` output to distinguish dir creation from template copy; (3) reconsider whether missing hook templates should be WARN (advisory) rather than FAIL; (4) thread the user's task into NEXT_RUN; (5) tolerate or pre-handle `Cargo.lock`. The external fixture is NOT committed to the AKAR repo; the AKAR commit contains only the dogfood report, CHANGELOG, and version bump. AKAR source was not modified. Cargo.toml bumped to v0.24.0 after verification passed.
